# Lapdev 项目需求文档 - 用户故事与验收标准

## 1. 概述
本文档从用户视角出发，以用户故事（User Story）的形式描述 Lapdev 的核心功能，并为每个故事定义可验证的验收标准（Acceptance Criteria）。

## 2. 用户角色
- **个人开发者**：使用 Lapdev 进行日常编码，希望获得高效的 AI 辅助。
- **团队负责人**：为团队搭建开发环境，关注成本、数据隐私与部署便捷性。
- **社区贡献者**：为 Lapdev 开发 Skill 或贡献代码。

## 3. 用户故事与验收标准

### 3.1 核心 IDE 功能

**US-01: 作为个人开发者，我希望能在浏览器中看到一个完整的文件树，以便导航我的项目文件。**
- **验收标准**：
  - 打开工作区后，文件树正确显示项目根目录下的所有文件和文件夹（遵循 .gitignore 规则）。
  - 文件树支持折叠/展开文件夹。
  - 文件树实时更新：当文件在外部被创建、删除或重命名时，文件树自动刷新。
  - 可以通过点击文件在编辑器中打开文件。

**US-02: 作为个人开发者，我希望有一个功能丰富的代码编辑器，以便高效编写代码。**
- **验收标准**：
  - 支持语法高亮、括号匹配、代码折叠。
  - 支持多光标编辑。
  - 支持在多个已打开文件之间快速切换（Tab 标签栏）。
  - 提供至少两种主题（浅色/深色），并且可以即时切换。

**US-03: 作为个人开发者，我希望在 IDE 中直接使用终端，以便运行命令和脚本。**
- **验收标准**：
  - 可以在 IDE 底部或侧边打开一个完整的终端（xterm.js）。
  - 终端连接到服务器上的真实 Shell 环境（bash/zsh）。
  - 支持同时打开多个终端 Tab。
  - 输入的命令能正常执行并返回结果，延迟可接受（<50ms）。

**US-04: 作为个人开发者，我希望 IDE 能显示 Git 状态，并支持基本操作，以便进行版本控制。**
- **验收标准**：
  - 在文件树和编辑器边栏中显示文件的 Git 状态（已修改、新增、未跟踪）。
  - 可以查看文件的差异对比（可视化 diff）。
  - 支持将文件暂存（stage）、提交（commit）和切换分支等基本操作。

**US-05: 作为个人开发者，我希望获得代码智能提示，以便提高编码效率。**
- **验收标准**：
  - 打开 Rust 或 TypeScript 文件时，能自动提供补全建议。
  - 鼠标悬停在代码上时，显示类型和文档提示。
  - 可以跳转到定义、查找引用。
  - 实时显示代码中的错误和警告（波浪线）。

### 3.2 AI 功能（BYOK）

**US-06: 作为个人开发者，我希望在设置页面填入我的第三方 API Key，以便使用自己的 AI 模型。**
- **验收标准**：
  - 提供一个清晰易用的设置界面。
  - 支持填入 API Key、自定义 Base URL、选择模型名称。
  - 配置可以保存，并在当前会话中生效。API Key 不暴露在前端日志中。
  - 可以配置多个模型，并快速切换当前使用的模型。

**US-07: 作为个人开发者，我希望有一个 AI 聊天面板，可以直接与 AI 对话并引用我的代码。**
- **验收标准**：
  - 可以在侧边栏打开 AI 聊天面板。
  - 对话支持流式输出（逐字显示）。
  - 可以手动引用当前打开的文件或选中的代码片段作为上下文。
  - AI 能基于上下文回答问题或生成代码。

**US-08: 作为个人开发者，我希望能获得 AI 内联代码补全建议，以便更快完成代码行。**
- **验收标准**：
  - 在编辑器中编写代码时，可以触发 AI 补全建议。
  - 建议以幽灵文本形式显示在当前光标后。
  - 按 Tab 键接受建议，按 Esc 键取消。
  - 该功能仅在用户配置了有效的 API Key 后可用。

**US-09: 作为个人开发者，我希望在 Agent 模式下，AI 能自动执行任务，但重要操作需我确认。**
- **验收标准**：
  - 在 AI 面板中可以开启 Agent 模式。
  - AI 可以读取文件、搜索代码库并生成修改方案。
  - 任何写入文件的操作在执行前必须弹窗请求用户确认。
  - 用户可以一键批准或拒绝所有修改。

### 3.3 Skill 系统

**US-10: 作为社区贡献者，我希望按照规范编写一个 .skill.md 文件，以扩展 Lapdev 的能力。**
- **验收标准**：
  - 项目提供清晰的 Skill 开发文档，说明 .skill.md 的元数据格式和指令编写规范。
  - 将编写好的 .skill.md 放入全局或项目级 skills 目录后，Lapdev 启动时应自动加载。
  - 通过 CLI 命令 `lapdev skill install <name>` 可以从市场安装 Skill。

**US-11: 作为个人开发者，我希望 AI 能根据我的问题自动匹配并使用已安装的 Skill。**
- **验收标准**：
  - 当用户提问与某个 Skill 的描述高度相关时，AI 的回答应遵循该 Skill 的指令。
  - 可以在 AI 面板中看到当前被激活的 Skill 名称。

### 3.4 BMAD 敏捷方法论支持

**US-12: 作为个人开发者，我希望新建项目后就能直接使用 BMAD 工作流，无需手动配置。**
- **验收标准**：
  - 在 Lapdev 中新建或打开一个项目时，系统应自动检测 BMAD 是否已安装。
  - 若未安装，系统应提示用户并询问是否启用 BMAD。
  - 用户确认后，系统优先尝试在线安装 (`npx bmad-method install`)。
  - 若在线安装成功，项目下出现完整的 `_bmad` 目录，并可使用所有 BMAD 工作流。

**US-13: 作为个人开发者，当在线安装 BMAD 失败时，我希望系统能提供一个可用的简化版，使工作流不中断。**
- **验收标准**：
  - 若在线安装失败或 Node.js 不可用，系统应自动将内置简化版文件写入项目。
  - 简化版至少包含：一个快速工作流（quick-flow）、开发者（developer）和产品经理（pm）智能体。
  - 安装后，在 AI 面板中可以看到 `bmad-quick-flow`、`bmad-developer`、`bmad-pm` 三个可用技能。
  - 用户可以随时尝试重新在线安装完整版。

### 3.5 部署与国内环境适配

**US-14: 作为团队负责人，我希望使用 Podman 快速部署 Lapdev，并且能使用国内镜像源加速。**
- **验收标准**：
  - 提供一份详尽的 Podman 部署指南。
  - 项目提供 `podman-compose.yml` 文件，可以通过一条命令启动 Lapdev 服务。
  - 指南中明确介绍如何配置国内镜像源（如阿里云）以加速基础镜像拉取。
  - Lapdev 官方镜像推送到国内 registry（如阿里云 ACR），并提供离线镜像包下载。

**US-15: 作为个人开发者，我希望从国内能流畅地获取代码和参与社区。**
- **验收标准**：
  - Lapdev 的主代码仓库托管在 Gitee（码云）上，国内访问稳定。
  - Gitee 仓库会自动同步到 GitHub，保持国际可见度。
  - 项目在 Gitee 上接受 Issue 和 Pull Request。

---

## 4. 整体验收标准
在项目交付时，应达到以下整体目标：
1. 在无网络环境下，通过 Podman 启动 Lapdev，新建项目后可自动启用内置 BMAD 工作流。
2. 在有网络环境下，打开任意项目，执行在线安装后可无缝切换到完整 BMAD。
3. 用户填入自己的 OpenAI / DeepSeek API Key 后，可正常进行 AI 对话和内联补全。
4. 文件树、编辑器、终端、Git 面板基础功能可用且无明显 Bug。
5. 所有代码在 Gitee 公开，遵循 MIT 协议，并自动同步至 GitHub 镜像。
